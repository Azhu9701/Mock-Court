# 万民幡 Rust 版 UI 设计

> 这不是聊天界面——这是多视角同时碰撞的观测窗口。

---

## 一、为什么聊天界面不行

聊天界面假设「一轮一轮对话」：使用者说一句，系统回一句。万民幡的根本模式是**多个独立视角同时展开推理**，它们的价值不在回合制的你来我往，而在并行产出的碰撞。

| 聊天界面 | 万民幡需要 |
|---------|----------|
| 单列时间线 | 多列并行流 |
| 一问一答 | 一问多答 + 综合 |
| 使用者主导节奏 | 魂各自推进，交叉追问自动触发 |
| 历史靠往上翻 | 每个魂有独立时间轴、修正历史、盲区记录 |
| 输入框是唯一入口 | 魂的状态、知识库检索、历史回溯都是入口 |

## 六、信息架构原则

1. **多列并流** — 魂各自独立空间，不轮播、不折叠到单列
2. **碰撞实时弹出** — 不在事后报告里藏，在发生时就可见
3. **综合持续更新** — 不是等待后的结果，是进展中的状态
4. **魂状态一等公民** — 修正历史、盲区、待审提案是 UI 元素，不是隐藏在 YAML 里的元数据
5. **移动端独立架构** — 不是桌面的缩小版，是标签页 + badge 的信息架构
6. **一切可回溯** — 知识库检索、历史合议回看、魂修正时间线，使用者总可以回到之前的任意状态
7. **计算成本透明** — 每次合议标注 LLM 调用次数和估计 token 消耗，使用者在合议前就能看到

## 七、技术栈：Rust + Next.js

│       └── 通过 API Routes SSE 接收魂 token   │
│                                             │

### 为什么 Next.js 适合这个场景

| Next.js 特性 | 万民幡怎么用 |
|-------------|------------|
| **React Server Components** | 魂列表、战绩统计、知识库浏览——这些页面数据从 Rust gRPC 取，在服务端渲染，零客户端 JS |
| **Streaming + Suspense** | 合议页外层是 RSC 骨架（魂配置、预设声明表单），内层是 `Suspense` 包裹的流式面板 |
| **API Routes + SSE** | `/api/confer/route.ts` 建立 gRPC stream 到 Rust，转成 SSE 推给浏览器。魂 token、碰撞事件、综合更新——全是 SSE 事件 |
| **React 组件模型** | 每个魂面板是一个 `<SoulPanel>` 组件，接收 SSE 事件流。交叉检测碰撞是 `<CollisionToast>`。综合面板是 `<SynthesisPanel>` |
| **响应式** | Tailwind CSS，桌面 `grid-cols-3`，移动 `tabs`，一套代码 |

### 数据流（合议页为例）

```
1. 使用者提交合议表单 → POST /api/confer
2. Next.js API Route → Rust gRPC Confer(stream)
3. Rust 并行 spawn 魂 → 魂 token 流 → gRPC stream
4. API Route 将 gRPC stream 转为 SSE：
     event: soul_token     data: {"soul":"马克思","token":"AI 驱动的..."}
     event: soul_token     data: {"soul":"费曼","token":"我们先定义..."}
     event: collision      data: {"from":"费曼","to":"马克思","content":"..."}
     event: soul_done      data: {"soul":"马克思","output_path":"..."}
     event: synthesis      data: {"consensus":[...],"divergence":[...],...}
5. 前端 EventSource 接收 → 更新各 SoulPanel 状态
```

### SSE 事件类型

```typescript
type ConferenceEvent =
  | { type: "soul_token";   soul: string; token: string }
  | { type: "soul_done";    soul: string; output_path: string }
  | { type: "collision";    from: string; to: string; content: string; injected: boolean }
  | { type: "synthesis";    consensus: string[]; divergence: string[]; blindspots: string[]; contradiction: string; actions: string[] }
  | { type: "cost";         llm_calls: number; tokens_used: number; estimated_cost: number }
  | { type: "error";        soul?: string; message: string }
  | { type: "done" }
```

### 流式渲染示意

```typescript
// app/confer/[id]/page.tsx (Client Component)
"use client";

export default function ConferPage({ params }: { params: { id: string } }) {
  const [panels, setPanels] = useState<Map<string, string[]>>(new Map());
  const [collisions, setCollisions] = useState<CollisionEvent[]>([]);
  const [synthesis, setSynthesis] = useState<Partial<Synthesis>>({});

  useEffect(() => {
    const es = new EventSource(`/api/confer?id=${params.id}`);
    es.addEventListener("soul_token", (e) => {
      const { soul, token } = JSON.parse(e.data);
      setPanels(prev => {
        const next = new Map(prev);
        next.set(soul, [...(next.get(soul) || []), token]);
        return next;
      });
    });
    es.addEventListener("collision", (e) => {
      setCollisions(prev => [...prev, JSON.parse(e.data)]);
    });
    es.addEventListener("synthesis", (e) => {
      setSynthesis(JSON.parse(e.data));
    });
    return () => es.close();
  }, [params.id]);

  return (
    <div className="grid grid-cols-3 gap-4 h-screen">
      {[...panels.entries()].map(([soul, tokens]) => (
        <SoulPanel key={soul} name={soul} tokens={tokens} />
      ))}
      <CollisionFeed events={collisions} />
      <SynthesisPanel data={synthesis} />
    </div>
  );
}
```

### TUI 保留为辅助界面

CLI 和 TUI（ratatui）仍然保留，和 Rust 核心库同进程运行，零序列化开销。适用于本地快速使用和脚本集成。共享同一套 `soul-banner-core` 库，只是不经过 gRPC。
