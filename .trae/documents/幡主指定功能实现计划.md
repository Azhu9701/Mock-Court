# 幡主指定功能实现计划

## 概述

当前幡主（审查官）通过环境变量 `AIONUI_REVIEWER_SOUL` 静态指定，默认为 `"未明子"`。用户无法在前端界面中选择或更改幡主。本功能允许用户在设置面板中从已炼化魂魄中选择幡主，并将其传递给 `analyzeTask` API，覆盖环境变量的默认值。

---

## 实现步骤

### 步骤 1：后端 — AnalyzeRequest 增加 reviewer 字段

**文件**: `rust/api/src/routes/possess.rs`

- 在 `AnalyzeRequest` 结构体中增加 `reviewer: Option<String>` 字段
- 修改 `analyze_task` 函数中幡主名称的获取逻辑：
  - 优先使用请求中的 `reviewer` 参数
  - 若未提供，回退到环境变量 `AIONUI_REVIEWER_SOUL`
  - 若环境变量也未设置，回退到默认值 `"未明子"`

**变更位置**: `possess.rs` L96（AnalyzeRequest 结构体）、L340（reviewer_name 获取逻辑）

---

### 步骤 2：前端 API 层 — analyzeTask 支持 reviewer 参数

**文件**: `nextjs/lib/api.ts`

- 修改 `analyzeTask` 函数签名，增加可选的 `reviewer?: string` 参数
- 在 POST body 中传递 `reviewer` 字段

**变更位置**: `api.ts` L378-L385

---

### 步骤 3：前端 — 设置面板增加幡主选择器

**文件**: `nextjs/components/settings-dialog.tsx`

- 将静态的"幡主"说明文字替换为交互式下拉选择器
- 加载所有可用魂魄列表（复用已有的 `fetchSouls` API）
- 用户选择一个魂作为幡主
- 选择结果保存到 `localStorage`（key: `aionui-banner-lord`）
- 显示当前幡主名称
- 尚无幡主概念需要新增到后端持久化——选择仅在前端 localStorage 存储，每次调用 analyzeTask 时传递

**变更位置**: `settings-dialog.tsx` L592-L598

---

### 步骤 4：前端 — PossessionEntry 传递幡主选择

**文件**: `nextjs/components/possession-entry.tsx`

- 在 `onStart` 函数中从 `localStorage` 读取用户选择的幡主名称
- 调用 `analyzeTask` 时传递 `reviewer` 参数
- 确保 UI 中的审查官名称显示正确

**变更位置**: `possession-entry.tsx` L70-L160（onStart 函数）、L378-L385（analyzeTask 调用处）

---

## 数据流

```
用户选择幡主（settings-dialog）
  → localStorage.setItem("aionui-banner-lord", "列宁")
  
附体流程启动（possession-entry）
  → localStorage.getItem("aionui-banner-lord") → "列宁"
  → analyzeTask(task, reviewer="列宁")
  
后端 analyze_task
  → body.reviewer ?? env("AIONUI_REVIEWER_SOUL") ?? "未明子"
  → spawn_banner_lord_review(reviewer_name="列宁", ...)
```

---

## 回退策略

- 若用户未在设置中选择幡主：`localStorage` 中无值 → 前端不传 `reviewer` 参数 → 后端使用环境变量默认值
- 若 localStorage 中的幡主名称在后端 soul registry 中不存在：后端 `get_soul` 返回 Err → 回退到 "审查官不可用，默认通过"
- 向后兼容：不传 `reviewer` 参数时行为与当前完全一致

---

## 不需要修改的部分

- `spawn_banner_lord_review` 函数无需修改（已接受任意 SoulProfile）
- `StartPossessionRequest` 无需修改（幡主不参与附体会话）
- `presets.ts` 无需修改
- 模型路由无需修改（幡主审查始终使用高推理能力模型）
